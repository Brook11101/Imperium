# 部署方案与非功能设计

## 1. 部署目标

部署方案需要满足以下目标：

- 本地开发环境可一键启动
- Docker Compose 单机环境可完整跑通 MVP
- OpenClaw CLI 运行环境可控
- 后续可平滑迁移到测试或生产环境

## 2. Docker 服务清单

建议 Compose 包含以下服务：

- `imperium-web`
- `imperium-api`
- `imperium-worker`
- `mysql`
- `rocketmq-namesrv`
- `rocketmq-broker`
- `rocketmq-dashboard`

## 3. OpenClaw 部署策略

当前阶段推荐：

- 将 OpenClaw CLI 与运行时依赖内置到 `imperium-worker` 镜像中

原因：

- Java Worker 通过 CLI 直接调用更简单
- 避免额外的 Runner 服务拆分
- Docker Compose 场景下运维更直接

后续若规模扩大，再考虑把 OpenClaw Runtime 拆成独立运行单元。

## 4. 容器职责

### `imperium-api`

- 提供业务 API
- 不直接执行 CLI
- 负责写库与 Outbox

### `imperium-worker`

- 消费 RocketMQ
- 调用 OpenClaw CLI
- 跑定时调度任务
- 写回执行结果与审计记录

### `imperium-web`

- 托管前端静态资源
- 通过 Nginx 反向代理后端 API

## 6. 非功能设计

### 6.1 可靠性

- 数据写入与事件投递采用 Outbox 保证一致性
- RocketMQ 消费必须幂等
- Agent CLI 调用失败可重试
- 停滞议案支持升级和回滚

### 6.2 可观测性

建议引入：

- Micrometer
- Prometheus
- Grafana
- Loki 或 ELK

至少需要监控：

- API 响应时间
- RocketMQ 消费积压
- Worker CLI 调用成功率
- 停滞议案数量
- override 数量
- 审计回流次数

### 6.3 安全性

MVP 建议：

- 单用户模式
- 管理接口受基础鉴权保护
- Agent 回调接口使用短期令牌
- 公开接口与内部接口严格分层

### 6.4 性能

建议目标：

- 普通查询响应小于 300ms
- 状态推进请求响应小于 500ms
- Agent 调度异步化，不阻塞请求线程
- 单议案内部有序，多议案并行

### 6.5 可维护性

- 按业务域拆分包结构
- Controller 不直接写复杂业务逻辑
- 所有状态流转必须经过统一状态服务
- 所有关键事件都必须进入时间线
- 所有 CLI 调用都必须写日志表

## 9. 当前技术取舍

### 为什么不是微服务

- 当前阶段业务复杂度集中在制度逻辑，不在服务拆分

### 为什么不是 BPM 引擎

- 当前流程规则强但规模仍可控，代码化状态机更简单直观

### 为什么不是同步 Agent 调用

- CLI 调用可能较慢，必须异步化以保护 API 响应时间

### 为什么是 MySQL + RocketMQ

- MySQL 负责稳定事实存储
- RocketMQ 负责异步驱动与扩展能力

## 10. 设计结论

`Imperium` 当前阶段推荐的部署与非功能基线是：

- Docker Compose 单机部署
- Java API + Java Worker 分离运行
- Worker 内置 OpenClaw CLI
- MySQL 8.4 负责事实存储
- RocketMQ 5.x 负责异步驱动
- 以可靠性、可观测性、可追踪性为 MVP 非功能底线
