# 领域模型与状态机实现

## 1. 设计目标

领域层的目标不是简单保存“任务数据”，而是用代码表达 `Imperium` 的制度规则。

因此领域设计必须保证：

- 核心对象有明确边界
- 状态流转受严格控制
- 运行模式能影响流转路径
- 权限边界可被程序校验
- 重要行为能产出领域事件

## 2. 核心领域对象

### 2.1 `Docket`

`Docket` 是主聚合根，表示一份议案卷宗。

负责承载：

- 当前状态
- 当前责任角色
- 运行模式
- 风险等级
- 原始法令信息
- 当前有效裁决与授权信息

### 2.2 `SenateSession`

表示一次元老院会话。

负责承载：

- 元老意见
- 共识点
- 争议点
- 推荐动议
- 会话开启与关闭时间

### 2.3 `TribuneReview`

表示保民官审查结果。

负责承载：

- 审查结论
- 否决或退回理由
- 补充说明

### 2.4 `CaesarDecision`

表示恺撒裁决结果。

负责承载：

- 批准 / 驳回 / 限制 / override
- 批注说明
- 约束条件

### 2.5 `Delegation`

表示执政官的派发记录。

负责承载：

- 目标角色
- 执行目标
- 依赖关系
- 执行状态

### 2.6 `AuditRecord`

表示审计记录。

负责承载：

- 风险结论
- 质量结论
- 回流原因

### 2.7 `ArchiveRecord`

表示归档记录。

负责承载：

- 最终摘要
- 产出清单
- 归档时间

## 3. 聚合边界建议

### 主聚合

- `Docket`

### 从属聚合或关联实体

- `SenateSession`
- `TribuneReview`
- `CaesarDecision`
- `Delegation`
- `AuditRecord`
- `ArchiveRecord`

建议原则：

- 所有状态变化必须通过 `Docket` 统一入口完成
- 不允许绕过 `Docket` 直接修改关键状态

## 4. 状态定义

与产品文档保持一致：

- `EdictIssued`
- `Triaged`
- `InSenate`
- `Debating`
- `VetoReview`
- `AwaitingCaesar`
- `Mandated`
- `Delegated`
- `InExecution`
- `UnderAudit`
- `Archived`
- `Rejected`
- `Suspended`
- `Revoked`

## 5. 状态机实现策略

### 5.1 方案选择

当前阶段推荐：

- 自研轻量状态机
- 采用 `Enum + Rule + Guard + Action` 组合实现

不建议当前阶段引入 Spring StateMachine。

原因：

- 当前状态机规模可控
- 规则表达更适合业务代码直写
- 可读性和可调试性更强

### 5.2 状态枚举

```java
public enum DocketState {
    EDICT_ISSUED,
    TRIAGED,
    IN_SENATE,
    DEBATING,
    VETO_REVIEW,
    AWAITING_CAESAR,
    MANDATED,
    DELEGATED,
    IN_EXECUTION,
    UNDER_AUDIT,
    ARCHIVED,
    REJECTED,
    SUSPENDED,
    REVOKED
}
```

### 5.3 合法流转规则

建议定义为统一规则表：

```java
Map<DocketState, Set<DocketState>> VALID_TRANSITIONS
```

并结合以下信息做校验：

- 当前角色
- 当前运行模式
- 当前权限
- 当前 guard 条件

## 6. 运行模式与状态机映射

### `StandardSenate`

- 必须经过元老院
- 必须经过保民官
- 必须经过恺撒裁决
- 审计允许回流

### `DirectDecree`

- 可跳过元老院
- 快速进入恺撒确认或授权
- 仍保留审计阶段

### `TribuneLock`

- 必须经过元老院与保民官
- 审计后更容易回流到恺撒
- 适合高风险事务

## 7. 状态责任人规则

建议将“谁能推进哪个状态”独立成一层规则：

- `praeco`：`EdictIssued -> Triaged`
- `senate engine`：`InSenate -> Debating`
- `tribune`：`Debating/VetoReview -> AwaitingCaesar/InSenate/Rejected`
- `Caesar`：`AwaitingCaesar -> Mandated/InSenate/Revoked`
- `consul`：`Mandated -> Delegated -> InExecution`
- `praetor/scriba`：`UnderAudit -> Archived/InExecution/AwaitingCaesar`

## 8. 权限与状态机协同

权限校验不应只依赖 Agent allow list，还要结合状态。

例如：

- 即便 `consul` 可以调执行角色，也不能在 `AwaitingCaesar` 状态提前派发
- 即便 `tribune` 可以做审查，也不能在 `InExecution` 状态直接重写执行结果

因此需要：

- 角色权限校验
- 状态责任人校验
- 运行模式校验

三层共同生效。

## 9. 领域事件设计

建议每次重要状态变化都产出领域事件，例如：

- `DocketCreated`
- `DocketTriaged`
- `SenateDebateOpened`
- `SenateDebateCompleted`
- `TribuneReviewSubmitted`
- `CaesarDecisionMade`
- `DelegationCreated`
- `ExecutionProgressReported`
- `AuditReturned`
- `DocketArchived`

## 10. 事务边界建议

### 同步事务内完成

- 核心状态变化
- 当前快照更新
- 时间线写入
- Outbox 写入

### 异步事务处理

- 元老院任务派发
- OpenClaw CLI 调用
- 调度恢复任务
- 归档后置处理

## 11. 设计结论

领域层的职责是把产品制度规则固化成可执行代码。

一句话总结：

- `Imperium` 的领域模型不是“任务管理模型”，而是“制度驱动的议案生命周期模型”。
