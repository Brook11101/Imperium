# 前后端接口与页面技术方案

## 1. 设计目标

前后端方案需要同时满足两类目标：

- 在产品层体现恺撒、元老院、执政官、案卷等治理感
- 在工程层保证页面清晰、接口稳定、扩展可控

## 2. 前端技术基线

建议采用：

- React 19
- TypeScript 5.x
- Vite 6
- Ant Design 5
- Zustand
- TanStack Query
- React Router

选择原因：

- 现代主流前端栈
- 类型约束好，适合中后台复杂状态
- Ant Design 适合快速搭建治理型后台界面
- TanStack Query 适合处理轮询、缓存和列表刷新

## 3. 页面结构

### `/`

对应：`Caesar Desk`

用于：

- 发布法令
- 查看待恺撒裁决事项
- 查看全局告警
- 查看活跃议案

### `/senate/:docketId`

对应：`Senate Chamber`

用于：

- 查看元老院意见
- 查看共识与争议点
- 查看保民官意见

### `/mandates`

对应：`Mandate Board`

用于：

- 查看所有议案状态分布
- 按模式、优先级、风险筛选

### `/dispatch/:docketId`

对应：`Consul Dispatch`

用于：

- 查看授权事项
- 查看分派与执行进度
- 处理重试与升级

### `/audit`

对应：`Audit and Veto`

用于：

- 查看高风险事项
- 查看否决记录
- 查看 override 记录

### `/archives`

对应：`Archives`

用于：

- 查看归档案卷
- 查看最终摘要、时间线、产出物

## 4. 前端状态管理策略

### 4.1 Zustand

负责：

- 当前页面 UI 状态
- 筛选条件
- 当前选中议案
- 弹窗和局部交互状态

### 4.2 TanStack Query

负责：

- 服务端数据获取
- 缓存
- 列表刷新
- 轮询
- 失效重拉

## 5. API 分层

### 5.1 对前端公开 API

用于：

- 查询议案
- 提交裁决
- 查看看板
- 查看档案
- 查看活动流

### 5.2 对 Agent 的内部 API

用于：

- 进度回传
- 结果回传
- 元老意见回传
- 审计回传

内部 API 需要：

- `callbackToken`
- 内部访问限制
- HMAC 或短期令牌校验

## 6. 接口设计方向

### 6.1 议案接口

- `POST /api/dockets`
- `GET /api/dockets`
- `GET /api/dockets/{id}`
- `POST /api/dockets/{id}/suspend`
- `POST /api/dockets/{id}/revoke`

### 6.2 元老院接口

- `GET /api/dockets/{id}/senate`
- `POST /api/dockets/{id}/senate/redeliberate`

### 6.3 保民官接口

- `POST /api/dockets/{id}/tribune/approve`
- `POST /api/dockets/{id}/tribune/reject`
- `POST /api/dockets/{id}/tribune/return`

### 6.4 恺撒接口

- `POST /api/dockets/{id}/caesar/approve`
- `POST /api/dockets/{id}/caesar/reject`
- `POST /api/dockets/{id}/caesar/override`
- `POST /api/dockets/{id}/caesar/restrict`

### 6.5 调度接口

- `POST /api/dockets/{id}/delegate`
- `POST /api/dockets/{id}/retry`
- `POST /api/dockets/{id}/escalate`

### 6.6 活动流接口

- `GET /api/dockets/{id}/timeline`

### 6.7 看板接口

- `GET /api/dashboard/overview`
- `GET /api/dashboard/alerts`

### 6.8 档案接口

- `GET /api/archives`
- `GET /api/archives/{id}`

## 7. 页面与接口映射

### Caesar Desk

需要接口：

- 新建法令
- 待裁决列表
- 高风险议案列表
- 活跃执行概览
- 最近归档列表

### Senate Chamber

需要接口：

- 议案详情
- 元老意见
- 共识点与争议点
- 保民官意见
- 重议动作

### Mandate Board

需要接口：

- 议案分页列表
- 状态统计
- 风险统计
- 当前责任角色过滤

### Consul Dispatch

需要接口：

- 授权详情
- 分派列表
- 执行项列表
- 阻塞项
- 重试 / 升级操作

### Audit and Veto

需要接口：

- 高风险议案
- 否决记录
- override 记录
- 审计回流事项

### Archives

需要接口：

- 归档列表
- 案卷详情
- 完整时间线
- 产物清单

## 8. 实时性方案

MVP 建议优先采用：

- 普通页面轮询
- 活动流与告警可考虑 SSE

原因：

- 实现复杂度低于 WebSocket 全栈方案
- 足够支撑当前治理型中后台场景

## 9. 接口响应规范

建议统一响应结构：

- `code`
- `message`
- `data`
- `traceId`

错误场景统一输出：

- 业务错误码
- 可读错误说明
- 追踪 ID

## 10. 设计结论

前后端方案的目标不是堆叠页面与接口，而是把制度流程变成可理解、可操作、可追踪的产品体验。
