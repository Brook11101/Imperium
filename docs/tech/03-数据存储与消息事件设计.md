# 数据存储与消息事件设计

## 1. 数据设计原则

数据层设计遵循以下原则：

- MySQL 是系统事实源
- RocketMQ 是异步事件驱动层，不是事实源
- 核心状态采用关系模型存储
- 审计、附加属性和扩展信息允许 JSON 字段承载
- 保留快照 + 历史事件双结构
- 不采用纯 Event Sourcing

## 3. 索引策略

建议优先建立以下索引：

- `docket(state, current_owner, updated_at)`
- `docket(mode, state)`
- `docket_event(docket_id, created_at)`
- `senate_opinion(session_id, agent_id)`
- `delegation(docket_id, status)`
- `execution_task(docket_id, status)`
- `outbox_event(publish_status, created_at)`
- `agent_call_log(docket_id, role_code, started_at)`

## 4. RocketMQ 设计目标

RocketMQ 主要用于：

- 解耦主请求与异步执行
- 驱动元老院审议任务
- 驱动 Agent 调用任务
- 驱动停滞恢复与升级逻辑

RocketMQ 不用于：

- 存储最终事实状态
- 替代数据库审计能力

## 5. Topic 设计

建议控制在 3 个 Topic 内，避免过度细分。

### `imperium-domain-topic`

用于领域状态事件。

Tag 建议：

- `DOCKET_CREATED`
- `STATE_CHANGED`
- `CAESAR_DECISION_MADE`
- `TRIBUNE_REVIEW_MADE`
- `AUDIT_RETURNED`
- `DOCKET_ARCHIVED`

### `imperium-agent-job-topic`

用于 Agent 执行任务。

Tag 建议：

- `PRAECO_JOB`
- `SENATE_JOB`
- `TRIBUNE_JOB`
- `CONSUL_JOB`
- `EXECUTION_JOB`
- `SCRIBE_JOB`

### `imperium-recovery-topic`

用于恢复与升级任务。

Tag 建议：

- `STALL_SCAN`
- `RETRY_DISPATCH`
- `ESCALATE_TO_CONSUL`
- `ESCALATE_TO_TRIBUNE`
- `ESCALATE_TO_CAESAR`
- `ROLLBACK_TO_SNAPSHOT`

## 6. 顺序与并行策略

建议使用 `docketId` 作为消息分片键。

原则如下：

- 单个议案内关键事件有序
- 不同议案之间可并行

## 7. 一致性方案

采用：

- 本地事务 + Outbox 模式

流程如下：

1. 业务请求事务内写业务数据
2. 同事务写入 `outbox_event`
3. Outbox Publisher 扫描未发布事件
4. 发送 RocketMQ
5. 成功后标记已发布

这样可避免：

- 数据库成功但消息没发出去
- 消息发出去但数据库没提交

## 8. 幂等设计

消费者端必须幂等。

建议方式：

- 每条消息带 `eventId`
- 消费端记录处理记录
- 遇到重复消息直接返回成功

## 9. 历史记录策略

系统必须保留两类历史：

- 结构化事件历史：`docket_event`
- 可读活动流：由事件和业务数据聚合生成的 timeline

## 10. 设计结论

数据与消息层的核心原则是：

- MySQL 负责真实状态
- RocketMQ 负责异步驱动
- Outbox 负责一致性
- 时间线负责可审计与可解释性
