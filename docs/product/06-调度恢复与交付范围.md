# 调度恢复与交付范围

## 1. 调度设计目标

`Imperium` 既要有自动调度能力，也不能破坏制度边界。

因此调度系统的目标不是“尽快把事情做完”，而是：

- 在合法状态之间推动事务前进
- 在停滞时提供恢复机制
- 在风险上升时升级到更高权力层
- 保留完整调度记录

## 2. 调度类型

### 2.1 状态调度

- 由状态变化触发
- 根据当前状态唤起对应角色

### 2.2 执行调度

- 由执政官派发触发
- 负责把授权事务拆成执行项

### 2.3 恢复调度

- 由停滞、超时或人工干预触发
- 用于重试、升级与回滚

## 3. 调度元数据建议

每个议案建议保留如下字段：

```json
{
  "enabled": true,
  "lastProgressAt": "",
  "retryCount": 0,
  "escalationLevel": 0,
  "lastDispatchStatus": "success",
  "snapshot": {
    "state": "Delegated",
    "savedAt": ""
  }
}
```

## 4. 停滞恢复建议

推荐恢复顺序如下：

1. 自动重试派发
2. 升级给 `consul`
3. 如果形成僵局或风险上升，升级给 `tribune`
4. 如果议案陷入高争议，则回到 `AwaitingCaesar`
5. 必要时回滚到最近稳定状态

## 5. 接口方向

当前阶段不定义详细实现协议，但建议资源模型围绕 `dockets`。

### 5.1 基础接口

- `POST /api/dockets`
- `GET /api/dockets`
- `GET /api/dockets/{id}`
- `POST /api/dockets/{id}/transition`
- `POST /api/dockets/{id}/suspend`
- `POST /api/dockets/{id}/revoke`

### 5.2 元老院接口

- `POST /api/dockets/{id}/senate/open`
- `GET /api/dockets/{id}/senate`
- `POST /api/dockets/{id}/senate/redeliberate`

### 5.3 保民官接口

- `POST /api/dockets/{id}/tribune/approve`
- `POST /api/dockets/{id}/tribune/reject`
- `POST /api/dockets/{id}/tribune/return`

### 5.4 恺撒接口

- `POST /api/dockets/{id}/caesar/approve`
- `POST /api/dockets/{id}/caesar/reject`
- `POST /api/dockets/{id}/caesar/override`
- `POST /api/dockets/{id}/caesar/restrict`

### 5.5 执行接口

- `POST /api/dockets/{id}/delegate`
- `POST /api/dockets/{id}/execution/progress`
- `POST /api/dockets/{id}/execution/complete`

### 5.6 审计与归档接口

- `POST /api/dockets/{id}/audit/pass`
- `POST /api/dockets/{id}/audit/return`
- `POST /api/dockets/{id}/archive`

## 6. MVP 范围

### 6.1 本阶段必须交付

- 完整角色模型
- 完整状态模型
- 三种运行模式
- 恺撒案头
- 元老院议场
- 议案总看板
- 执政官调度台
- 保民官审查能力
- 恺撒裁决能力
- 审计到归档闭环
- 统一活动流

### 6.2 本阶段暂不交付

- `CampaignMode` 的完整实现
- 高级多 Governor 联邦体系
- 复杂投票机制
- 移动端优先体验
- 全角色绩效评分系统
- 深度历史档案检索能力

## 7. 路线图

### Phase 1：制度闭环

- 角色体系
- 状态机
- 运行模式
- 议案页面
- 保民官审查
- 恺撒裁决
- 调度闭环
- 归档闭环

### Phase 2：治理增强

- 监察官能力
- 更强的 override 审计
- 预算异常机制
- 更强的恢复与升级机制

### Phase 3：帝国扩张

- `CampaignMode`
- Governor 生态
- 外部系统适配
- 更深的知识归档能力

## 8. 风险

### 8.1 主要风险

- 流程过重
- 元老角色区分不明显
- 保民官失去真实阻断力
- 恺撒过度 override
- 执行层越权影响政治层

### 8.2 缓解方向

- 通过运行模式控制治理强度
- 为元老角色提供明确差异化职责
- 把否决与回流写入制度硬规则
- 明确记录并高亮 override
- 用权限矩阵与状态机共同约束角色

## 9. 当前阶段验收标准

当前产品定义阶段，在以下条件满足时视为完成：

- 产品定位稳定且自洽
- 角色体系清晰
- 状态机完整
- 权限矩阵可解释且可执行
- 运行模式有明确边界
- 页面结构与数据模型可作为下一阶段输入
- MVP 范围清楚，不与后续阶段混淆

## 10. 本文档结论

当前阶段的目标不是把实现方案定死，而是把 `Imperium` 的制度逻辑、交付边界与下一阶段输入定义清楚。
