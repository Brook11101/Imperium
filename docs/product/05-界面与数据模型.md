# 界面与数据模型

## 1. 页面设计目标

页面设计必须服务于治理感，而不仅是任务可视化。

用户在系统中应感受到：

- 自己在发布法令
- 元老院正在处理议案
- 国家机器正在执行
- 案卷正在形成历史记录

## 2. 关键页面

### 2.1 `Caesar Desk`

用途：

- 主入口
- 发布法令
- 查看待裁决议案
- 观察全局摘要和关键告警

建议模块：

- 新法令输入区
- 待恺撒裁决列表
- 正在议政的议案
- 正在执行的授权事项
- 关键风险告警
- 最近归档案卷

### 2.2 `Senate Chamber`

用途：

- 展示结构化元老院辩论

建议模块：

- 议题摘要
- 元老观点区
- 共识点
- 争议点
- 保民官评注
- 呈送恺撒的动议摘要

### 2.3 `Mandate Board`

用途：

- 全局议案看板

建议能力：

- 按状态分列
- 按模式、优先级、风险筛选
- 查看当前责任角色
- 高亮停滞、阻塞和等待恺撒的议案

### 2.4 `Consul Dispatch`

用途：

- 执政官调度中心

建议能力：

- 查看已授权议案
- 拆分执行项
- 监控执行进度
- 查看阻塞与依赖关系
- 发起重试与升级

### 2.5 `Audit and Veto`

用途：

- 风险控制与制度监督中心

建议能力：

- 查看高风险议案
- 查看否决记录
- 查看审计异常
- 查看 override 历史

### 2.6 `Archives`

用途：

- 帝国案卷档案库

建议能力：

- 浏览已归档议案
- 查看最终摘要与交付物
- 查看完整时间线
- 查看争议与裁决历史

## 3. 核心业务对象

`Imperium` 的主业务对象建议命名为 `docket`，即议案卷宗。

它比普通 task 更强调：

- 有正式法令来源
- 有辩论记录
- 有否决与裁决记录
- 有执行与审计记录
- 最终形成案卷

## 5. 关键子结构

### 5.1 元老意见

```json
{
  "agent": "senator_juris",
  "stance": "object",
  "summary": "Current proposal exceeds declared authority",
  "details": "Detailed rationale",
  "generatedAt": "2026-03-29T10:10:00Z"
}
```

### 5.2 分派项

```json
{
  "role": "legatus",
  "objective": "Implement auth backend",
  "status": "in_progress",
  "dependsOn": [],
  "assignedAt": "2026-03-29T11:00:00Z"
}
```

### 5.3 活动记录

```json
{
  "at": "2026-03-29T10:15:00Z",
  "kind": "debate",
  "actor": "senator_fiscus",
  "title": "Fiscal Objection",
  "summary": "The full scope is too expensive for first pass",
  "details": "Recommend staged rollout"

```

## 6. 活动流模型

建议将所有关键行为统一进一条时间线。

活动类型建议包括：

- `state`
- `debate`
- `review`
- `decision`
- `delegation`
- `execution`
- `audit`
- `archive`
- `override`
- `alert`
- `tool_result`

## 7. 设计结论

页面结构和数据模型都必须围绕“议案如何在制度中被处理”展开，而不是简单复用任务管理工具语义。
