# 权限矩阵与核心流程

## 1. 权限设计目标

权限矩阵的目标不是做安全装饰，而是塑造制度结构。

`Imperium` 的权限设计必须确保：

- 议政权与执行权分离
- 否决权真实存在
- 执行层不能反向篡改政治层
- 书记官与监察官保持相对中立

## 2. 权限原则

### 2.1 接旨层只负责受理

`praeco` 可以分流，但不能直接决定高价值事务的命运。

### 2.2 元老层负责辩论，不直接执行

元老层的职责是形成意见，不是直接调动执行机构。

### 2.3 保民官负责阻断，不负责落地

`tribune` 可以挡、退、升，但不应成为执行指挥者。

### 2.4 执政官负责派发，不重写议案本意

`consul` 的任务是把批准议案转成执行命令，而不是重新立法。

### 2.5 执行层只落实，不改制度

执行角色可以上报阻塞与结果，但不应随意修改政治层决策。

### 2.6 书记官负责记载，不主导执行

`scriba` 是总结终点，不是新调度中心。

### 2.7 监察官只监督，不经营日常流程

`censor` 应对制度提出审视，不应深入日常操作链。

## 3. 推荐权限矩阵

### `praeco`

- 可调用：`senator_strategos`、`senator_juris`、`senator_fiscus`、`consul`
- 不可直接调用：`tribune` 以下执行角色

### `senator_strategos`

- 可调用：`tribune`、`scriba`、`consul`

### `senator_juris`

- 可调用：`tribune`、`scriba`、`consul`

### `senator_fiscus`

- 可调用：`tribune`、`scriba`、`consul`

### `tribune`

- 可调用：`scriba`、`consul`
- 可通过工作流退回元老院

### `consul`

- 可调用：`legatus`、`praetor`、`aedile`、`quaestor`、`scriba`、`governor_*`

### `legatus`

- 可调用：`consul`、`scriba`

### `praetor`

- 可调用：`consul`、`scriba`、`tribune`

### `aedile`

- 可调用：`consul`、`scriba`

### `quaestor`

- 可调用：`consul`、`scriba`、`tribune`

### `scriba`

- 可调用：`censor`
- 默认不再向执行层发起调度

### `censor`

- 可调用：`scriba`

### `governor_*`

- 可调用：`consul`、`scriba`

## 4. 恺撒特权

虽然 `Caesar` 不作为普通 Agent 存在，但在产品层拥有最高权限：

- 覆盖保民官决定
- 要求重新议政
- 撤销已经授权的议案
- 暂停任意非终态议案
- 以军令模式强制推进
- 对批准议案附加执行限制

所有恺撒特权都必须进入案卷记录。

## 5. 制度硬规则

以下规则建议写入产品宪法，并在后续实现中严格约束：

- 执行层不得直接向恺撒请命
- 元老层不得直接指挥执行层
- 保民官不得接管执行工作
- 书记官不得转化为隐性调度中心
- 监察官不得成为日常执行推动者

## 6. 核心业务流程

### 6.1 标准议政流程

1. 恺撒下达法令
2. `praeco` 规范化处理
3. 元老院进入辩论
4. `tribune` 审查
5. 恺撒裁决
6. `consul` 派发执行
7. 执行结果进入审计
8. `scriba` 归档

### 6.2 军令直达流程

1. 恺撒下达直接法令
2. `praeco` 判断为低风险或紧急事务
3. 议案快速进入恺撒确认或直接授权
4. `consul` 派发执行
5. 执行后仍进入审计归档

### 6.3 否决回流流程

1. 元老辩论形成动议
2. `tribune` 识别严重问题
3. 议案退回元老院
4. 元老在新约束下重议

### 6.4 审计回流流程

1. 执行完成
2. 审计发现风险、质量或结果偏差
3. 议案回流到执行层整改
4. 若问题带有战略性争议，则升级回恺撒裁决

## 7. 本文档结论

`Imperium` 的权限矩阵不是为了限制交流本身，而是为了把“谁能提案、谁能阻断、谁能拍板、谁能执行、谁来记账”区分清楚。
