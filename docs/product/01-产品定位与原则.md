# 产品定位与原则

## 1. 产品定义

`Imperium` 是一个以古罗马制度为隐喻的多 Agent 治理系统。

它的核心不是让多个 Agent 自由协作，而是把复杂事务放入一套正式制度中处理：

- 恺撒下令
- 传令官受理
- 元老院辩论
- 保民官审查
- 恺撒裁决
- 执政官派发
- 专职机构执行
- 审计归档

从产品类型上看，`Imperium` 不是单纯的 Agent workflow 工具，而是一个“治理型操作系统”。

## 2. 核心命题

多数 Agent 产品强调的是自动化与自主性，`Imperium` 强调的是治理能力。

具体命题如下：

- 复杂事务不应直接从输入跳到执行
- 高价值决策应当先经过结构化分歧
- 否决权必须真实有效
- 决策权与执行权应被分离
- 从法令到归档的全过程必须可见、可审计、可追溯

## 3. 要解决的问题

当前常见 Agent 系统存在这些问题：

- Agent 权责边界模糊
- 规划与执行混在一起
- 反对意见缺失，系统天然倾向于“顺着做”
- 用户只能看到结果，看不到形成过程
- 高风险事务缺少制度性挑战与阻断
- 完成后很难形成组织级知识沉淀

`Imperium` 试图解决的不是“怎么再快一点”，而是“怎么在复杂任务中形成可信决策”。

## 4. 产品愿景

让一个用户以恺撒身份，治理一套 AI 国家机器，使每个重要事务都能被正式讨论、审查、授权、执行并归档。

## 5. 产品目标

### 5.1 本阶段目标

- 建立稳定的罗马制度产品叙事
- 定义核心角色体系
- 定义完整状态机与运行模式
- 定义权限矩阵和关键流程
- 定义界面结构与核心数据模型
- 定义 MVP 边界，为下一阶段设计与实现提供输入

### 5.2 中长期目标

- 让复杂任务先经过“制度化决策”，再进入执行
- 形成可治理、可追责、可复盘的 Agent 协作系统
- 让系统逐步沉淀为组织级决策与执行档案库

## 6. 非目标

以下内容不属于当前产品目标：

- 完整历史拟真的古罗马模拟器
- 纯叙事化角色扮演产品
- 完全自由通信的 Agent 网络
- 只追求极限执行速度的系统
- 只有世界观包装、没有制度约束的“换皮产品”

## 7. 目标用户

### 7.1 核心用户

- 独立开发者
- 技术创业者
- 工程负责人
- AI 工作流设计者

### 7.2 次级用户

- AI 治理研究团队
- 复杂决策型产品团队
- 需要“人类在环”的自动化系统使用者

## 8. 产品价值

`Imperium` 的核心价值不在于把任务自动做完，而在于把任务处理过程变得可信。

主要价值体现在：

- 通过制度化角色划分，减少职责混乱
- 通过元老院辩论，让不同视角成为正式输入
- 通过保民官否决，防止高风险方案直接落地
- 通过恺撒裁决，让用户始终保留最终主权
- 通过完整案卷，让每个决策都能沉淀为组织记忆

## 9. 产品原则

### 9.1 制度先于自主

系统必须先建立制度边界，再允许 Agent 在边界内发挥。

### 9.2 先议后行

复杂事务优先进入辩论和审查，不直接默认执行。

### 9.3 否决权不是装饰

保民官的角色必须能阻断、退回或升级事务。

### 9.4 恺撒保留主权

争议事务最终应回到用户裁决，而不是由系统自动结束。

### 9.5 全程留痕

每个关键节点都应进入活动流与案卷。

### 9.6 权力分层明确

产品必须明确区分：

- 接旨层
- 议政层
- 否决层
- 裁决层
- 行政调度层
- 执行层
- 审计层
- 归档层

## 10. 本文档结论

`Imperium` 不是“多个 AI 自由聊天”，而是一套“恺撒统摄下的制度化 AI 治理系统”。

后续所有设计都必须服务于这一定位。
